# App Data Contract

The iOS app now reads from a single `SandFestPayload` instead of screen-local sample arrays.

## Current Source

- Processed bootstrap source: `data/processed/app-bootstrap.json`
- Bundled seed generated from source: `ios/TexasSandFest/Resources/sandfest-seed.json`
- Swift loader: `ios/TexasSandFest/AppDataStore.swift`
- Swift model: `ios/TexasSandFest/Models.swift`

This keeps the app shippable offline while we wait for live access to Eventeny, QuickBooks, vendor files, sponsor records, volunteer rosters, maps, and operating documents.

The web app uses separate fail-closed projections generated by `lib/public-bootstrap.mjs` and `lib/public-media-manifest.mjs`. `public/data/app-bootstrap.json` and `GET /api/public/bootstrap` contain only governed event-guide facts, public schedule entries, privacy-minimized zones, and public alerts. The public media manifest contains browser-safe asset metadata without local paths or upstream fetch details. These records can be reopened from browser cache if beach cell service drops.

`POST /api/public/concierge` composes answers from those projections plus the public ticket, sponsor, vendor, weather, ferry, and camera contracts. It does not read internal processed bootstrap collections, persist question text, or cache responses. Its citations allow only in-page anchors and reviewed HTTPS sources, and deployment verification rejects unexpected or private response fields.

The shared bootstrap payload also includes `alert`, an inactive-by-default emergency alert record. Web reads the live value from `/api/public/alert`. Native iOS loads its bundled payload immediately, refreshes the complete privacy-minimized `/api/public/bootstrap` projection at app launch or on manual refresh, and preserves the bundled value when connectivity or validation fails.

The native cache contains only that public projection. It is bound to the exact API origin and bundled event ID, so a local board response, stale annual event, or response from another environment cannot be reopened as production data. The response must contain non-empty, uniquely identified schedule and zone collections; an active alert must include the public audience. An explicitly labeled `board_demo` response may retain the richer synthetic schedule overlay, while production responses replace the public schedule with the governed API collection.

## Payload Areas

`data/processed/app-bootstrap.json` is an internal normalization document. It is not a public API or deployable artifact.

- `guide`: event identity, dates, location, update timestamp
- `alert`: public emergency alert state, severity, message, audience, update timestamp, and optional expiration
- `schedule`: customer schedule and staff operational milestones
- `zones`: beach map areas and crowd/ops status
- `ticketOptions`: Eventeny ticket handoff cards
- `ticketProducts`: Stripe-ready product catalog for GA, VIP, sponsor, and review-gated raffle paths
- `sponsors`: sponsor tier, invoice, fulfillment, and next action state
- `vendors`: vendor status, category, and booth assignment
- `coverage`: volunteer coverage by zone
- `financeSignals`: QuickBooks and finance readiness indicators

The public projection allows only `guide`, visitor-approved `schedule` entries, zones without operational `status`, and `alert`. Staff schedule entries, `ticketOptions`, sponsor and vendor workflow, volunteer coverage, finance signals, publishing identity, local file paths, and ingestion manifests must remain on authenticated operations surfaces. The board runtime may add only a bounded synthetic-data label when the server is explicitly running in board-demo mode.

## Ingestion Rule

Every future import should normalize into this payload or an explicit extension of it before the app consumes the data. That gives us one reviewable boundary between chaotic source files and the customer/admin product.

Run this after changing the processed bootstrap payload:

```bash
npm run ios:seed
```

Public web data and ticket products are synced separately:

```bash
npm run public:sync
```

## Backend Contract

Authenticated clients can receive the internal normalized shape through a role-governed endpoint:

```text
GET /api/app/bootstrap
Authorization: Bearer <staff-or-public-token>
```

Customer clients use only the projected public bootstrap. Admin clients receive staff-only fields based on role.

The current native staff screens still use bundled demonstration collections.
Connecting those screens to `GET /api/app/bootstrap` requires the native OIDC
session and role enforcement; no staff-only fields are cached by the public
launch refresh.
